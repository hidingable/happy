# Happy Coder 核心业务逻辑

## 1. 端到端加密体系

### 设计理念

Happy 的核心安全承诺：**服务器永远看不到用户的明文数据**。所有会话内容、消息、设备信息、Artifacts 都在客户端加密后才传输。

### 密钥层级

```
用户账户密钥对（非对称，TweetNaCl/libsodium）
  │
  ├── Master Key（主密钥）
  │     │
  │     ├── Session DEK（会话数据加密密钥）
  │     │     └── 加密会话消息内容
  │     │
  │     ├── Machine DEK（设备数据加密密钥）
  │     │     └── 加密设备元数据
  │     │
  │     └── Artifact DEK（Artifact 数据加密密钥）
  │           └── 加密 Artifact 内容
  │
  └── DEK 本身用 Master Key 加密后存储在服务器
```

### 关键文件

- **App 端加密原语**: `happy-app/sources/encryption/`（AES、HMAC-SHA512、密钥派生）
- **App 端业务加密**: `happy-app/sources/sync/encryption/`（Session/Machine/Artifact 加密）
- **CLI 端加密**: `happy-cli/src/api/encryption.ts`（TweetNaCl）
- **服务端认证**: `happy-server/sources/app/auth/auth.ts`（privacy-kit Token）

## 2. 会话（Session）生命周期

### 创建会话

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  手机 App │         │  Server  │         │ CLI      │
│          │         │          │         │ Daemon   │
└────┬─────┘         └────┬─────┘         └────┬─────┘
     │                    │                    │
     │ 1. 创建会话请求     │                    │
     │ （加密 metadata）   │                    │
     │───────────────────→│                    │
     │                    │ 2. Socket 通知      │
     │                    │───────────────────→│
     │                    │                    │ 3. 选择 Agent
     │                    │                    │   (Claude/Codex/...)
     │                    │                    │
     │                    │                    │ 4. 启动 Agent 进程
     │                    │←───────────────────│
     │ 5. 实时消息推送     │                    │ 5. 流式消息
     │←───────────────────│                    │   （加密后上传）
     │                    │                    │
```

### 双模式运行

CLI 支持两种模式，可实时切换：

| 模式 | 触发方式 | Agent 控制方 | 输出目标 |
|------|---------|-------------|---------|
| **本地模式** | 终端直接运行 `happy` | 本地 PTY 终端 | 终端 + 远程同步 |
| **远程模式** | 手机 App 创建会话 | SDK 驱动 | 手机 App |

关键代码: `happy-cli/src/claude/loop.ts`

## 3. Daemon 系统

CLI 运行时自动启动后台 Daemon 进程，Daemon 的核心职责：

```
┌──────────────────────────────────┐
│           Happy Daemon           │
├──────────────────────────────────┤
│ ✅ 维护与 Server 的 WebSocket    │
│ ✅ 接收远程会话创建请求           │
│ ✅ 管理多个并发 Agent 会话        │
│ ✅ 自动版本检测与升级             │
│ ✅ 会话状态同步                  │
│ ✅ 机器信息上报                  │
└──────────────────────────────────┘
```

关键代码: `happy-cli/src/daemon/`（有独立 CLAUDE.md 架构文档）

## 4. 多 Agent 集成

Happy CLI 不绑定单一 AI Agent，支持多种后端：

| Agent | 目录 | 集成方式 |
|-------|------|---------|
| **Claude Code** | `happy-cli/src/claude/` | `@anthropic-ai/claude-code` SDK + PTY |
| **OpenAI Codex** | `happy-cli/src/codex/` | CLI 包装 |
| **Google Gemini** | `happy-cli/src/gemini/` | ACP (Agent Communication Protocol) |
| **OpenClaw** | `happy-cli/src/openclaw/` | 自定义集成 |

用户通过命令行参数选择：`happy`（默认 Claude）/ `happy codex` / `happy gemini`

## 5. 数据同步引擎（InvalidateSync）

App 使用自研的 `InvalidateSync` 模式协调多数据源同步：

```
┌─────────────────────────────────────┐
│          Sync 单例                   │
├─────────────────────────────────────┤
│ syncSessions()     → 会话列表同步    │
│ syncMessages()     → 消息同步        │
│ syncMachines()     → 设备同步        │
│ syncSettings()     → 设置同步        │
│ syncProfile()      → 个人资料同步    │
│ syncPurchases()    → 订阅状态同步    │
│ syncArtifacts()    → Artifact 同步   │
│ syncFriends()      → 好友同步        │
│ syncFeed()         → 动态同步        │
│ syncOTA()          → OTA 更新检查    │
├─────────────────────────────────────┤
│ 每个 sync 方法独立调度              │
│ 失败自动重试                        │
│ Socket.IO 事件触发增量同步          │
└─────────────────────────────────────┘
```

关键文件:
- `happy-app/sources/sync/sync.ts` — Sync 主类
- `happy-app/sources/sync/storage.ts` — Zustand Store

## 6. QR 码认证流程

```
1. CLI 生成密钥对 → 创建 TerminalAuthRequest → 获得 auth_id
2. auth_id 编码为 QR 码 → 在终端展示
3. 手机 App 扫描 QR 码 → 获取 auth_id
4. App 向 Server 发送认证响应（POST /v1/auth/response）
5. Server 将认证信息返回给 CLI
6. CLI 获得持久 Token，后续自动连接
```

关键文件:
- `happy-cli/src/api/auth.ts`
- `happy-cli/src/ui/qr.ts`
- `happy-server/sources/app/api/routes/authRoutes.ts`
